# 助手使用偏好

最后更新：2026-02-10

## 语言

- 默认响应语言：中文。
- 代码标识符保持英文。

## 沟通风格

- 回答直接、实用。
- 相比抽象讨论，更偏好清晰的分步骤执行。
- 解释代码时，包含文件路径引用。

## 编码风格偏好

- 新增或修改代码时，添加详细注释，提升可读性。
- 优先给出可在当前仓库直接运行的示例。
- 代码改动与现有项目结构保持一致。
- 涉及 `examples` 的改动时，同步更新 `examples/README.md`（目录、命令、注意事项）。

## 学习型输出

- 学习任务优先提供清单与具体示例。
- 制作学习计划时，优先使用“按天任务”形式。

## 验证

- 代码改动后，尽可能执行格式化与测试。
- 清晰汇报验证命令与结果。

## 通用任务产出规则（可复用）

- 当用户要求“生成某周/某功能内容”时，默认按“代码 + 测试 + 文档 + 验证”四件套交付。
- 每周内容需要提供独立可运行入口：新增 `examples/weekXX/cmd/main.go`，支持 `go run ./examples/weekXX/cmd`。
- 代码层面：补齐路由、handler、store（如适用）及错误码映射（至少覆盖 400/404/500 的可判定分支）。
- 测试层面：至少覆盖成功路径、资源不存在、参数非法三类场景；测试应可稳定重复运行。
- 学习文档层面：新增对应 `notes/weekXX.md`，使用按天打卡结构（目标、每日任务、验收、复盘模板）。
- 接口文档层面：新增或更新 `docs/weekXX-*.md`，明确输入、输出、错误码和示例。
- 手动验证层面：提供可直接执行的 `curl` 命令，并包含成功与失败示例。
- 运行文档层面：每周若新增示例或入口，必须同步更新 `examples/README.md`（目录、命令、示例输出）。
- 一致性规则：功能代码变更与文档变更必须同次提交完成，避免文档过期。
- 质量门禁：完成后执行格式化与 `go test ./...`，并在回复中汇报关键结果。
